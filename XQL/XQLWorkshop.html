WHAT ARE XDR DATASETS?

Dataset Definition

XDR datasets are structured collections of security data that provide the foundation for advanced threat detection, investigation, and response capabilities within Cortex XDR.

Key Characteristics

  • Standardized Schema - Consistent data structure across sources
  • Real-time Updates - Continuously refreshed security intelligence
  • Rich Context - Detailed metadata and relationships
  • Queryable via XQL - Powerful analysis capabilities

Dataset Categories

  • Core Security - Findings, cases, issues
  • Asset Management - Inventory and metadata
  • Vulnerability Data - Risk assessments and scoring
  • Cloud Security - DSPM, CIEM, CWP data

Why Datasets Matter

Datasets enable correlation across security domains, providing comprehensive visibility into your security posture and enabling data-driven decision making.

CORE SECURITY DATASETS

Foundation of Security Operations

These datasets form the backbone of security incident management and response workflows.

findings

Primary security findings dataset - Contains security findings from various sources, providing a centralized view of potential risks across your environment.

findings_revisions

Finding change tracking - Tracks revisions and updates made to findings over time, enabling audit trails and trend analysis.

cases

Security case management - Stores information about security cases created to manage and investigate related findings.

issues

Issue tracking system - Contains information about issues created to track and manage related findings and remediation efforts.

ASSET MANAGEMENT DATASETS

Comprehensive Asset Visibility

These datasets provide complete visibility into your organization's digital assets and their security posture.

asset_inventory

Complete asset catalog - Comprehensive inventory of all assets in your environment including servers, containers, cloud resources, and network devices.

issue_to_asset

Issue-asset relationships - Maps security issues to the specific assets they are associated with, enabling targeted remediation.

issue_to_finding

Issue-finding correlation - Maps security issues to their corresponding findings, providing traceability and context.

case_to_issue

Case-issue mapping - Maps security cases to the issues they contain, supporting hierarchical incident management.

VULNERABILITY MANAGEMENT DATASETS

Risk-Based Vulnerability Intelligence

Advanced vulnerability data with context, scoring, and prioritization capabilities.

uvm_findings

Unified Vulnerability Management - Contains comprehensive vulnerability findings with CVSS scores, exploitability data, and risk context from the UVM module.

scans_management

Scan orchestration data - Contains information about security scan management, scheduling, and execution across your infrastructure.

cwp_packages_raw

Cloud Workload Protection packages - Raw data about packages and software components in your Cloud Workload Protection environment for vulnerability analysis.

Key Use Cases

  • Risk-based vulnerability prioritization
  • Exploitability assessment and EPSS scoring
  • Container and workload vulnerability tracking
  • Compliance and audit reporting

cas_policies

DATASET RELATIONSHIPS & JOINS

Understanding Data Connections

Effective XQL queries often require joining multiple datasets to provide comprehensive security insights.

Common Join Patterns

  • findings ← → asset_inventory
    Link security findings to specific assets
  • uvm_findings ← → asset_inventory
    Connect vulnerability data to assets
  • issues ← → findings
    Group related findings into issues
  • cases ← → issues
    Organize issues into security cases

Key Join Fields

  • asset_id - Primary asset identifier
  • finding_id - Unique finding identifier
  • issue_id - Issue tracking identifier
  • case_id - Security case identifier
  • scan_id - Scan execution identifier

Pro Tip

Always use LEFT JOIN when some records might not have matches, and consider dataset size for performance optimization.

BEST PRACTICES FOR DATASET USAGE

Performance Optimization

  • Filter Early - Apply filters before joins
  • Limit Timeframes - Use time-based filtering
  • Select Specific Fields - Avoid SELECT *
  • Use Appropriate Joins - Inner vs Left joins

Query Design

  • Start Small - Test with limited data first
  • Incremental Builds - Add complexity gradually
  • Document Logic - Comment complex queries
  • Error Handling - Plan for missing data

Security Considerations

  • Understand data sensitivity levels
  • Apply appropriate access controls
  • Consider data retention policies
  • Implement audit logging

dataset = asset_inventory
| filter asset_type = "Container" // Filter early
| fields asset_id, asset_name // Select specific fields
| join type = left (dataset = uvm_findings) // Use appropriate join

CORE XQL STAGES

Dataset

Specifies the data source for your query.

dataset = asset_inventory

Filter

Narrows down the dataset by applying conditional checks to fields.

| filter xdm.asset.type.category = "Container Image"

Fields

Selects and renames fields to be included in the output.

| fields xdm.asset.name as Name

Alter

Creates new fields or modifies existing ones using expressions.

| alter risk_score = 100

CORE XQL STAGES (CONTINUED)

Bin

Groups records into specific time intervals.

| bin first_observed span = 1D

Join

Combines data from two or more datasets based on a common field.

| join (dataset = uvm_findings)

Comp

Computes aggregate values like counts, sums, or averages.

| comp count(asset_id) as total_assets by asset_type

Sort

Orders the final output in ascending or descending order.

| sort desc first_seen

ESSENTIAL FUNCTIONS

arrayindex

Extracts an element from an array based on its index.

arrayindex(split(to_string(first_observed), " "), 0)

split

Divides a string into an array of substrings using a delimiter.

split(cvss_severity, "_")

to_string / to_number

Converts data from one type to another.

to_string(first_observed)

add / multiply / divide

Performs basic mathematical operations.

add(multiply(UltraCriticalExploited, 100), multiply(UltraHighExploited, 50))

if

A conditional function that returns one value if a condition is true, and another if it's false.

if(Malware_Count >= 1 and risk_score < 90, 90, risk_score)

PART 1: INITIAL DATA AND FILTERING

We'll begin by selecting our base dataset and preparing it for analysis by filtering and selecting key fields.

Tasks

  • Begin the query by specifying the dataset: asset_inventory.
  • Filter the xdm.asset.type.category field to the value 'Container Image'.
  • Select and rename xdm.asset.provider to Cloud and xdm.asset.name to Name. Keep xdm.asset.first_observed, and xdm.asset.id, and xdm.asset.normalized_fields with their original names.
  • Extract the xdm.development_cycle_stages value from the nested xdm.asset.normalized_fields JSON and create a new field named sdlc.
  • Filter out any results where the sdlc field is null.

PART 1: SOLUTION

dataset = asset_inventory
| filter xdm.asset.type.category = "Container Image"
| fields xdm.asset.provider as Cloud, xdm.asset.name as Name, xdm.asset.first_observed as first_observed, xdm.asset.id as asset_id, xdm.asset.normalized_fields
| alter sdlc = xdm.asset.normalized_fields -> ["xdm.development_cycle_stages"][0]
| filter sdlc != null

PART 2: TIME GROUPING & DASHBOARD FILTERS

Next, we refine our data for dashboard use by formatting dates and adding dynamic filters.

Tasks

  • Aggregate all results that were first observed on the same day into a single group using a 1-day span.
  • Transform the full date and timestamp of first_observed into a simplified YYYY-MM-DD format and store it in a new field called first_seen. (Hint: Use the to_string, split, and arrayindex functions.)
  • Apply dashboard filters to the first_seen field, using the dynamic value $first_detected, and to the sdlc field, using the dynamic value $sdlc_stage.
  • Select the final fields to be carried forward: Name, first_seen, sdlc, and asset_id.
  • Order the results by the first_seen field in descending order.

PART 2: SOLUTION

| bin first_observed span = 1D
| alter first_seen = arrayindex(split(to_string(first_observed), " "), 0)
| filter first_seen = $first_detected
| filter sdlc = $sdlc_stage
| fields Name, first_seen, sdlc, asset_id
| sort desc first_seen

PART 3: JOINING WITH VULNERABILITY DATA

10:00

This is a large, multi-step join operation. We combine our data with vulnerability findings and then calculate a comprehensive risk score.

Tasks

  • Begin a join with the uvm_findings dataset.
  • Within the join, filter asset_category to 'Container Image' and select necessary fields.
  • Normalize cvss_severity to a simple severity field.
  • Count and classify vulnerabilities by severity and exploitability. Use the comp command to group findings and create separate count fields for each vulnerability category.
  • Calculate weighted temporal scores and final risk scores.

PART 3: SOLUTION

| join (dataset = uvm_findings
| filter asset_category = "Container Image"
| fields asset_name, cvss_severity, asset_id, exploitable, exploit_level, epss_score
| alter severity = arrayindex(split(cvss_severity, "_"), 2)
| comp
    count(if(severity = "Critical" and exploitable = true and to_number(epss_score) >= 0.75, 1, null)) as UltraCriticalExploited,
    count(if(severity = "High" and exploitable = true and to_number(epss_score) >= 0.75, 1, null)) as UltraHighExploited
by asset_name, asset_id
| alter temporal_score = add(multiply(UltraCriticalExploited, 100), multiply(UltraHighExploited, 50))
| alter risk_score = round(multiply(100, divide(temporal_score, add(temporal_score, 100))))
| filter risk_score > 0
| fields risk_score, asset_id
) as risk risk.asset_id = asset_id

PART 4: FINAL JOIN & MALWARE SCORING

10:00

Finally, we join with malware findings and make final adjustments to the score.

Tasks

  • Perform a left join with the findings dataset.
  • Filter for malware findings and count them per asset.
  • Set the final score: if malware is present and risk_score is under 90, set it to 90.
  • Create boolean malware flag and apply dashboard filters.

PART 4: SOLUTION

| join type = left (dataset = findings
| fields xdm.finding.category as Category, xdm.finding.asset_id
| filter Category contains "MALWARE"
| comp count(Category) as Malware_Count by xdm.finding.asset_id
) as findings findings.xdm.finding.asset_id = asset_id
| alter score = if(Malware_Count >= 1 and risk_score < 90, 90, risk_score)
| alter malware = if(Malware_Count >= 1, "true", "false")
| filter malware in ($malware_finding)
| fields Name, first_seen, sdlc, score, malware, asset_id

COMPLETE XQL QUERY

Full Query Implementation

Here's the complete query combining all the components we've analyzed: asset inventory, vulnerability scoring, malware detection, and risk enhancement.

dataset = asset_inventory
| filter xdm.asset.type.category = "Container Image"
| fields xdm.asset.provider as Cloud, xdm.asset.name as Name, xdm.asset.first_observed, xdm.asset.id as asset_id
| alter sdlc = xdm.asset.normalized_fields -> ["xdm.development_cycle_stages"][0]
| join (dataset = uvm_findings | comp count() as vuln_count by asset_id) as vulns
| join type = left (dataset = findings | filter xdm.finding.category contains "MALWARE") as malware
| alter final_score = if(malware != null, 90, risk_score)
| fields Name, sdlc, final_score, asset_id

KUBERNETES DATA SOURCES

Runtime Workload Data

  • xdr_data - Live Kubernetes workloads
  • action_k8s_workload - Pod runtime information
  • k8s_container_name - Container identification
  • k8s_namespace - Namespace context
  • k8s_pod_name - Pod identification

Image Vulnerability Data

  • cloud_resource_metadata - Image metadata
  • container_image_name - Image references
  • container_vulnerabilities - CVE data
  • cvss_severity - Risk classifications
  • exploitability_score - Threat intelligence

Key Challenge: Correlation Complexity

Kubernetes environments present unique challenges for security analysis - runtime workloads must be correlated with static image vulnerability data, requiring sophisticated join operations and data normalization techniques.

PART 1: KUBERNETES WORKLOAD DISCOVERY

Start by discovering and cataloging all active Kubernetes workloads in your environment.

Tasks

  • Query the xdr_data dataset and filter for Kubernetes workload actions using action_k8s_workload = "WORKLOAD_STARTED".
  • Extract essential workload metadata: k8s_namespace, k8s_pod_name, k8s_container_name, and container_image_name.
  • Create a normalized image reference by removing registry prefixes and tags. Store this in a field called normalized_image.
  • Filter for workloads from the last 7 days using the _time field.
  • Group results by unique combinations of namespace, pod, and image to avoid duplicates.

Success Criteria

Your query should return a distinct list of active Kubernetes workloads with clean image references ready for vulnerability correlation.

PART 1: SOLUTION

dataset = xdr_data
| filter action_k8s_workload = "WORKLOAD_STARTED"
| filter _time > current_time() - 7d
| alter normalized_image = arrayindex(split(arrayindex(split(container_image_name, "/"), -1), ":"), 0)
| fields k8s_namespace, k8s_pod_name, k8s_container_name, container_image_name, normalized_image
| dedup k8s_namespace, k8s_pod_name, normalized_image

Key Techniques

  • Nested string manipulation to clean image names
  • Time-based filtering for recent workloads
  • Deduplication to ensure unique workload records

PART 2: IMAGE VULNERABILITY CORRELATION

Now correlate your active workloads with vulnerability data to identify security risks.

Tasks

  • Join your workload data with the cloud_resource_metadata dataset using normalized_image and container_image_name.
  • Further join with container_vulnerabilities using the asset_id field.
  • Extract vulnerability severity information and create severity counters: Critical, High, Medium, and Low.
  • Calculate exploitable vulnerability counts using the exploitability_score field (score > 0.1).
  • Group results by workload (namespace, pod, container) to aggregate vulnerability data per running workload.

Success Criteria

Each active Kubernetes workload should be enriched with comprehensive vulnerability counts and exploitability metrics.

PART 2: SOLUTION

| join type = left (
    dataset = cloud_resource_metadata
    | filter cloud_provider = "containers"
    | alter image_match = arrayindex(split(arrayindex(split(container_image_name, "/"), -1), ":"), 0)
) as img_meta on normalized_image = img_meta.image_match
| join type = left (
    dataset = container_vulnerabilities
    | alter severity = uppercase(arrayindex(split(cvss_severity, "_"), 0))
    | alter is_exploitable = if(exploitability_score > 0.1, 1, 0)
) as vuln on img_meta.asset_id = vuln.asset_id
| comp Critical = count(if(vuln.severity = "CRITICAL", 1, null)),
     High = count(if(vuln.severity = "HIGH", 1, null)),
     Exploitable = sum(vuln.is_exploitable)
     by k8s_namespace, k8s_pod_name, k8s_container_name

PART 3: RISK SCORING & PRIORITIZATION

Create a comprehensive risk scoring system for your Kubernetes workloads based on vulnerability data and runtime context.

Tasks

  • Calculate a base_risk_score using weighted vulnerability counts: Critical (×4), High (×2), Medium (×1), Low (×0.5).
  • Apply an exploitability multiplier: if exploitable vulnerabilities > 0, multiply base score by 1.5.
  • Add runtime context bonuses:
    • Privileged namespaces (kube-system, kube-public): +20 points
    • Production-like namespaces (prod, production): +15 points
  • Normalize the final risk_score to a 0-100 scale and create risk categories: High (≥80), Medium (50-79), Low (<50).
  • Sort results by risk score (descending) to prioritize the most critical workloads.

PART 3: SOLUTION

| alter base_risk_score = add(
    multiply(Critical, 4),
    multiply(High, 2),
    multiply(Medium, 1),
    multiply(Low, 0.5)
)
| alter exploitability_bonus = if(Exploitable > 0, multiply(base_risk_score, 0.5), 0)
| alter namespace_bonus = if(
    k8s_namespace in ("kube-system", "kube-public"), 20,
    if(k8s_namespace contains "prod", 15, 0)
)
| alter total_score = add(base_risk_score, exploitability_bonus, namespace_bonus)
| alter risk_score = min(divide(multiply(total_score, 100), 50), 100)
| alter risk_category = if(
    risk_score >= 80, "High",
    if(risk_score >= 50, "Medium", "Low")
)
| sort desc risk_score

COMPLETE KUBERNETES VULNERABILITY ANALYSIS

dataset = xdr_data
| filter action_k8s_workload = "WORKLOAD_STARTED" and _time > current_time() - 7d
| alter normalized_image = arrayindex(split(arrayindex(split(container_image_name, "/"), -1), ":"), 0)

    | alter image_match = arrayindex(split(arrayindex(split(container_image_name, "/"), -1), ":"), 0)

    dataset = container_vulnerabilities
    | alter severity = uppercase(arrayindex(split(cvss_severity, "_"), 0)),
           is_exploitable = if(exploitability_score > 0.1, 1, 0)
) as vuln on img.asset_id = vuln.asset_id
| comp Critical = count(if(vuln.severity = "CRITICAL", 1, null)),
     High = count(if(vuln.severity = "HIGH", 1, null)),
     Medium = count(if(vuln.severity = "MEDIUM", 1, null)),
     Low = count(if(vuln.severity = "LOW", 1, null)),
     Exploitable = sum(vuln.is_exploitable)
     by k8s_namespace, k8s_pod_name, k8s_container_name, normalized_image
